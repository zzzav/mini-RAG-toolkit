❓ ПРО ЧТО

Это 5-й урок стартового курса, где мы продолжаем собирать Telegram ИИ-секретаря не как набор разрозненных нод, а как систему со слоями.

В прошлом уроке бот уже научился ставить внутреннюю задачу в шину через Supabase / Postgres. Но логика всё ещё была: одно сообщение = одна отдельная задача.

В этом уроке добавляем слой краткосрочной памяти через Redis и переводим текстовый ввод в модель активного контекста.

Теперь система умеет:
— держать один живой контекст на пользователя
— обновлять одну и ту же внутреннюю задачу
— обновлять одну и ту же строку в Google Sheets
— завершать ожидание ввода либо по кнопке, либо позже по TTL

💡 ЧТО ВНУТРИ

— расширение Telegram Trigger: message + callback_query
— общая конфигурация входа и единый redis_key
— text-ветка с candidate_context_id и context_now
— чтение активного контекста через redis_get_context
— развилка if_context: new / existing
— prepare_new_context
— prepare_existing_context
— add_agent_text_task для создания новой записи в шине
— upd_agent_text_task для обновления существующей записи
— add_text_task по context_id, а не по message_id
— prepare_redis_context с key_data для Redis
— set_context с TTL
— send_context_msg с inline-кнопкой завершения ввода
— callback-ветка:
parse_callback
callback_cmd
get_context_for_check
if_context_active
finish_context

❕ ПОЧЕМУ ЭТО ВАЖНО

1️⃣ Мы отделяем краткосрочное состояние процесса от постоянного хранения

Google Sheets остаётся пользовательским слоем.

Supabase / Postgres остаётся внутренней шиной задач.

Redis берёт на себя именно короткую оперативную память.

2️⃣ Главным идентификатором текстового потока становится context_id

Теперь система работает не с отдельным message_id как с самостоятельной задачей, а с единым потоком мысли пользователя.

3️⃣ Один и тот же контекст обновляет:
— ту же внутреннюю задачу в t_course_agent_tasks
— ту же строку в пользовательской таблице

4️⃣ Мы подготавливаем правильный вход для следующего урока

После Redis и накопления контекста AI будет разбирать уже не обрывок вроде "и завтра утром", а полный накопленный пользовательский текст.

🔍 ОБРАТИ ВНИМАНИЕ

— почему redis_key вынесен в раннюю config-ноду
— почему candidate_context_id создаётся только в text-ветке
— чем отличается новый контекст от продолжения существующего
— почему в уроке часть стабильных данных берётся из более ранних нод, а не протаскивается по всем веткам
— что именно кладём в payload_json
— что именно кладём в key_data для Redis
— зачем текстовые поля в payload_json оборачиваются в JSON.stringify
— почему callback перед удалением ключа сначала сверяет актуальность context_id
— зачем merge_text ждёт завершения всех ключевых шагов перед ответом пользователю

📁 КАКОЙ МИНИМУМ ДАННЫХ ХРАНИМ

Во внутренней задаче:
— payload_json с накопленным текстом контекста
— context_id
— timestamps начала и последнего обновления

В Redis:
— context_id
— task_db_id
— chat_id
— text_raw_accumulated
— context_started_at
— context_last_input_at

То есть Redis здесь не хранит всё подряд, а только то, что нужно для реального процесса.

💎 КЛЮЧЕВАЯ МЫСЛЬ УРОКА

Redis в этой схеме — не ещё одна база на всякий случай.

Это слой памяти процесса, который помогает системе понять: это новый вход
или продолжение уже начатой мысли.

Теперь логика такая:
вход → проверка активного контекста → new/update → запись в системный слой → обновление памяти → ответ пользователю

🎓 ЧТО ДАЛЬШЕ

В следующем уроке подключим AI-слой:
— добавим транскрибацию голоса
— будем брать уже накопленный итоговый текст
— разберём его на структуру задачи
— и сохраним результат в result_json внутренней задачи

А в 7-м уроке вынесем этот анализ в отдельный workflow агента, добавим обработку протухших ключей и доведём всё до полноценной фиксации задач в пользовательском слое.
